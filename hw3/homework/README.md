# ДЗ — Hibernate, ACID и уровни изоляции транзакций

Самодостаточная реализация задания семинара 3 (не зависит от стартовых файлов в
корне `Seminar3`). Приложение на **Hibernate / JPA + HikariCP** прогоняет три
демонстрации по темам семинара и завершает работу.

## Что внутри

| Компонент | Описание |
|-----------|----------|
| **postgres** | PostgreSQL, БД `mydatabase`; схему создаёт сам Hibernate (`hbm2ddl=update`). На хост проброшен порт **5433** |
| **pgadmin** | pgAdmin с заранее зарегистрированной БД (http://127.0.0.1:5050) |
| **app** | Java-приложение: Hibernate ORM, ACID-транзакция, уровни изоляции |

## Три демонстрации

### 1. Hibernate ORM — сущности и связи
Маппинг сущностей и связи между ними:

- `Category` → `Product` — **One-to-Many** / **Many-to-One**;
- `Customer` → `Order` — **One-to-Many** / **Many-to-One**;
- `Order` ↔ `Product` — **Many-to-Many** (таблица-связка `order_product`).

Граф объектов сохраняется одной транзакцией (каскадом), затем читается через
`JOIN FETCH`; дополнительно показаны выборки через **JPQL** и **Criteria API**.

### 2. ACID — перевод денег в одной транзакции
Перевод между двумя счетами (`Account`) в рамках одной транзакции:

- **A (атомарность)** — при нехватке средств бросается исключение и делается
  `ROLLBACK`: оба счёта остаются нетронутыми;
- **C (согласованность)** — суммарный баланс не меняется;
- **D (долговечность)** — после `COMMIT` данные сохранены в БД;
- **I (изоляция)** — см. демо №3.

### 3. Уровни изоляции — неповторяемое чтение
Транзакция `T1` дважды читает один и тот же баланс, а между чтениями `T2`
меняет его и делает `COMMIT`:

| Уровень | Результат |
|---------|-----------|
| `READ COMMITTED` | второе чтение видит новое значение — **неповторяемое чтение** |
| `REPEATABLE READ` | оба чтения совпадают — T1 работает со снимком на момент старта |

> В PostgreSQL уровень `READ UNCOMMITTED` работает как `READ COMMITTED`
> (грязное чтение невозможно в принципе), поэтому показано именно различие
> между `READ COMMITTED` и `REPEATABLE READ`.

## Запуск

### Вариант 1 — всё в Docker (рекомендуется)

```bash
cd homework
docker compose up --build
```

Что произойдёт по порядку:
1. поднимется `postgres` (ждёт healthcheck);
2. соберётся и запустится `app`: Hibernate создаст схему и прогонит три демо;
3. контейнер `app` завершится с кодом 0 (это нормально — приложение разовое).

Ожидаемый вывод приложения:

```
=== 1. Hibernate ORM: сущности и связи ===
Сохранён граф: 2 категории, 2 товара, покупатель и его заказ.
Покупатель: Алексей
  Заказ от 2026-06-22:
    - Роман (500) [Книги]
    - Смартфон (30000) [Электроника]
JPQL — средняя цена товара: 15250.00
Criteria — найден товар: Смартфон, цена 30000

=== 2. ACID: перевод денег в одной транзакции ===
Старт:   Alice=1000, Bob=500
Перевод 300 (Alice → Bob): успех.
         Alice=700, Bob=800
Перевод 100000 (Alice → Bob): недостаточно средств → ROLLBACK.
         Alice=700, Bob=800  (баланс не изменился — атомарность)
Сумма балансов = 1500 (сохранена — согласованность)

=== 3. Уровни изоляции: неповторяемое чтение ===
--- READ COMMITTED ---
  T1: первое чтение      balance=1000
  T2: обновил баланс -> 2000 и сделал COMMIT
  T1: повторное чтение   balance=2000
  => неповторяемое чтение: значение изменилось внутри транзакции (1000 -> 2000)
--- REPEATABLE READ ---
  T1: первое чтение      balance=1000
  T2: обновил баланс -> 2000 и сделал COMMIT
  T1: повторное чтение   balance=1000
  => чтение повторяемо: оба раза 1000 (T1 видит снимок на момент старта)
```

### Вариант 2 — приложение с хоста

Поднять только БД:

```bash
cd homework
docker compose up -d postgres
```

Запустить приложение локально (по умолчанию ходит на `localhost:5433`):

```bash
cd app
./gradlew run
```

## pgAdmin

1. Открыть http://127.0.0.1:5050 (десктоп-режим, логин не спрашивается).
2. Слева в группе **Homework** уже есть сервер `mydatabase`.
3. При первом подключении ввести пароль `password`.

## Параметры подключения

| Параметр | Значение |
|----------|----------|
| Пользователь | `admin` |
| Пароль | `password` |
| БД | `mydatabase` |
| Хост (с хоста) | `localhost:5433` |
| Хост (внутри docker) | `postgres:5432` |

Адреса читаются из переменных окружения `DB_URL`, `DB_USER`, `DB_PASSWORD`,
поэтому один и тот же код работает и с хоста, и из контейнера.

## Очистка

```bash
docker compose down -v   # -v удалит том с данными БД
```
